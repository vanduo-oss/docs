Vanduo Ecosystem Guide

Vanduo is intentionally split into a lightweight core framework and optional canvas libraries. Everything is documented here on vanduo.dev, but you install only the packages your project needs — keeping bundle size and maintenance scope under control.

Why separate packages?

Most sites need buttons, forms, layout, and theme tooling — not hex terrain editors or flowchart builders. Shipping every visualization inside the framework would inflate every install.

  • Framework first: Start with @vanduo-oss/framework for UI, layout, and runtime components.
  • Add canvas libraries on demand: Install Hex Grid, Charts, or Flowchart only when a feature needs them.
  • Same docs site: Demos, API references, and guides for all packages live together so you can evaluate before installing.
  • Shared design language: Canvas packages read Vanduo --vd-* tokens and register with the same Vanduo.init lifecycle when loaded.

Vanduo Framework

The core UI system: responsive grid, typography, components (buttons, modals, tabs, forms), theme customizer, lazy loading, and the scoped runtime (Vanduo.init / Vanduo.destroy).

Choose when: building any Vanduo-powered site or app shell — always your first install.

pnpm add @vanduo-oss/framework

Hex Grid

Interactive flat-top hex canvas with pan/zoom, selection, terrain helpers, rotation, and axial coordinate math. Canvas-based, not SVG DOM nodes.

Choose when: strategy maps, tile prototypes, terrain editors, or experimental game mechanics.

Hex Grid docs →

pnpm add @vanduo-oss/hex-grid

Charts

Standalone SVG charting for dashboards: bar, line, area, scatter, pie, and donut charts. Declarative data-vd-chart markup or JavaScript API; theme-aware colors.

Choose when: analytics panels, KPI tiles, reports, or any data visualization that fits standard chart types.

Charts docs →

pnpm add @vanduo-oss/charts

Flowchart

SVG flow-diagram editor: draggable shapes, port-based arrows, double-click re-attachment, start/end marker caps, keyboard copy-paste, JSON import/export, and zoom/pan canvas.

Choose when: workflow builders, process maps, internal tooling, or documentation that needs editable diagrams.

Flowchart docs →

pnpm add @vanduo-oss/flowchart

How packages relate

vanduo.dev docs
├── @vanduo-oss/framework     ← UI, layout, theme, runtime (required base)
├── @vanduo-oss/hex-grid      ← optional canvas library
├── @vanduo-oss/charts        ← optional canvas library
└── @vanduo-oss/flowchart     ← optional canvas library

Canvas libraries are peers, not nested inside the framework bundle. Each ships its own CSS and IIFE/ESM build. Load the framework first, then add any canvas script and stylesheet your page needs.

<!-- Always -->
<link rel="stylesheet" href="/path/to/vanduo.min.css">
<script src="/path/to/vanduo.min.js"></script>

<!-- Only if this page uses charts -->
<link rel="stylesheet" href="/path/to/vanduo-charts.css">
<script src="/path/to/vanduo-charts.iife.js"></script>

<!-- Initialize the mounted subtree -->
<script>Vanduo.init(document.getElementById('app'));</script>

Extension packages call Vanduo.register(name, api) during load so Vanduo.init can auto-discover data-vd-chart, data-vd-flowchart, and similar markers. See Runtime Architecture for scoped init/destroy details.

Which package do I need?

Your goal Install Docs section
Landing page, admin UI, component library @vanduo-oss/framework Components
Dashboard with bar/line/pie charts framework + @vanduo-oss/charts Charts
Editable process or workflow diagram framework + @vanduo-oss/flowchart Flowchart
Hex tile map, strategy board, terrain tool framework + @vanduo-oss/hex-grid Hex Grid
All of the above on one site framework + each canvas package used on that route Load scripts per page or code-split by route

You can combine packages — for example, a dashboard page might use Charts inside Vanduo cards, while a separate admin route mounts Flowchart. Avoid loading every canvas bundle on every page; scope assets to the routes that need them.
