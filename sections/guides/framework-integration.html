Framework Integration Guide

Vanduo is designed to be completely framework-agnostic. Because it is built with pure CSS and Vanilla JavaScript modules, it can be dropped into any modern frontend framework (React, Vue, Nuxt, Svelte, etc.) seamlessly.

The Core Concept

When integrating Vanduo into a component-heavy framework, remember these two rules:

  1. CSS is Global: Import vanduo.css once in your global entry file (e.g., main.js, _app.js, or layout.vue).
  2. JS Needs the DOM: Vanduo's JavaScript components (like modals, dropdowns, and tabs) need elements to exist in the DOM before initialization. You must call Vanduo.init() after your framework has mounted the component to the screen.

Where to Import CSS

Vanduo's stylesheet should be imported once at the top level of your application. The exact file depends on your framework:

Framework Global Entry File
React (CRA) src/index.js or src/App.js
Next.js pages/_app.js or app/layout.tsx
Vue 3 src/main.js
Nuxt 3 nuxt.config.ts → css: ['vanduo/dist/vanduo.css']
Svelte / SvelteKit src/routes/+layout.svelte or src/app.html
Never import vanduo.css in individual components — it will duplicate styles and increase bundle size. One global import covers everything.

Handling SPA Route Changes

In a single-page application, navigating between routes replaces DOM elements without a full page reload. Vanduo's event listeners are attached to specific elements, so you need to re-initialize after each route change.

The pattern is the same in every framework: call Vanduo.init() in the mounting lifecycle of each page component that uses Vanduo interactive features (modals, dropdowns, tabs, tooltips, etc.).

Re-initialization in SPAs: When dealing with Client-Side Routing (like next/router or vue-router), navigating to a new page component replaces DOM elements. Ensure you call Vanduo.init() in the mounting phase of the new page component so Vanduo attaches its event listeners to the newly rendered elements!

Static content (grids, cards, typography, utilities) works without init() — only interactive JavaScript components need it.

SSR & SSG Considerations

When using Server-Side Rendering (Next.js, Nuxt, SvelteKit), Vanduo's CSS works perfectly — it's just a stylesheet and renders on the server without issues.

However, Vanduo's JavaScript requires the window and document APIs, which don't exist on the server. Follow these rules:

  • Never import Vanduo JS at the module top level in a server-rendered file. Use dynamic imports or guard with a browser check.
  • Always call Vanduo.init() inside a client-side lifecycle hook (useEffect, onMounted, onMount). These only run in the browser.
  • For Next.js App Router: mark the component that calls Vanduo.init() with 'use client'; at the top of the file.
'use client';

import { useEffect } from 'react';
import { Vanduo } from 'vanduo';

export default function VanduoProvider({ children }) {
  useEffect(() => {
    Vanduo.init();
    return () => Vanduo.destroy();
  }, []);

  return <>{children}</>;
}
