Troubleshooting & FAQ Guide

Stuck? This guide covers the most common issues when building with Vanduo — from missing components to dark mode quirks — and shows you exactly how to fix them.

#1 My navbar toggle doesn't work on mobile

Symptom: Tapping the hamburger icon on mobile does nothing — the menu doesn't expand.

Most likely cause: Vanduo.init() was not called, or it was called before the navbar existed in the DOM.

<!-- Make sure the navbar HTML is ABOVE the script tag -->
<nav class="vd-navbar">...</nav>

<!-- Then load and init Vanduo at the bottom of <body> -->
<script src="https://cdn.jsdelivr.net/gh/vanduo-oss/framework@v1.3.9/dist/vanduo.min.js"></script>
<script>Vanduo.init();</script>
</body>
SPA users: If you're using React, Vue, or another framework that swaps DOM content without page reloads, you must call Vanduo.init() after the navbar is rendered. See Framework Integration for lifecycle examples.

#2 Dark mode doesn't persist after refresh

Symptom: User toggles dark mode, reloads the page, and it reverts to light mode (or follows the OS).

Most likely cause: The data-theme attribute on <html> is set to "system" (the default), which means it always follows the OS preference on page load.

<!-- Option A: Set an explicit default so it doesn't flip back -->
<html lang="en" data-theme="dark">

<!-- Option B: Use the Theme Switcher component (saves to localStorage) -->
<button data-toggle="theme" aria-label="Toggle theme">
  <i class="ph ph-moon"></i>
</button>

How it works: The built-in data-toggle="theme" button saves the user's choice to localStorage under the key vanduo-theme-preference. On page load, Vanduo reads this key and applies data-theme="light" or data-theme="dark" automatically — but only if you included the Vanduo JS file in the head or early body, before the browser paints.

#3 Components don't work after SPA route change

Symptom: Navbar, dropdowns, modals, or tabs work on first page load but stop working after navigating to a new route in a React/Vue/Svelte app.

Root cause: Single-page apps replace DOM elements without a full page reload. Vanduo's event listeners were attached to the old DOM nodes, which no longer exist.

import { useEffect } from 'react';
import { Vanduo } from '@vanduo-oss/framework';

export default function MyPage() {
  useEffect(() => {
    Vanduo.init();
    return () => Vanduo.destroy();
  }, []);

  return (...);
}

The pattern is the same in every framework: call Vanduo.init() in the mounting lifecycle hook of each page component that uses Vanduo interactive features. Static content (grids, cards, typography) works without it — only JS-driven components need re-initialization.

#4 My custom theme colors aren't applying

Symptom: You added custom CSS with --color-primary: #ff0000; but buttons are still blue.

Most likely cause: Your custom CSS is loaded before Vanduo's CSS, so Vanduo's defaults override yours.

<!-- 1. Vanduo first -->
<link rel="stylesheet" href="vanduo.min.css">

<!-- 2. Your overrides second (wins the cascade) -->
<link rel="stylesheet" href="css/my-theme.css">

<!-- WRONG: this would be overridden by vanduo.min.css -->
<link rel="stylesheet" href="css/my-theme.css">
<link rel="stylesheet" href="vanduo.min.css">
Verify in DevTools: Open Chrome DevTools → Elements → select a button → Computed tab → filter for --color-primary. If it shows Vanduo's default, your CSS isn't loaded or isn't after Vanduo's.

#5 Icons aren't showing up

Symptom: You see empty squares or nothing where an icon should be.

Wrong

<i class="ph-home"></i>

Missing weight prefix. The icon font doesn't know which weight to use.

Right

<i class="ph ph-home"></i>

First ph = font family. Second ph-home = icon name + weight.

Bold

<i class="ph-fill ph-home"></i>

Use ph-fill for filled weight. Other options: ph-bold, ph-light, ph-thin, ph-duotone.

Also check: Did you include the Phosphor icon CSS? If you're using the full Vanduo bundle (vanduo.min.css), icons are included. If you're importing component CSS individually, add css/icons/icons.css.

#6 Cards don't have shadows in dark mode

Symptom: Cards look flat in dark mode — no visible shadow or depth.

This is by design: Vanduo's shadows use rgba() alpha blending. On dark backgrounds, low-alpha black shadows are nearly invisible. The framework compensates with slightly stronger borders and surface tints in dark mode.

Want more depth? You can boost shadows in dark mode by overriding the shadow variables in your custom CSS: --shadow-md: 0 8px 24px rgba(0,0,0,0.4);. Or switch cards to .vd-card-outlined for a crisp border that reads clearly in both modes.

#7 VanduoLazyLoad shows 'undefined'

Symptom: Your code references VanduoLazyLoad and throws undefined or is not defined.

Most likely causes:

  • You're running in a server-side rendering (SSR) context where window doesn't exist.
  • You imported the lazy-loader module but didn't include the main Vanduo JS bundle.
// Guard for SSR — only run in the browser
if (typeof window !== 'undefined' && window.VanduoLazyLoad) {
  window.VanduoLazyLoad.observe(element, (target) => {
    target.src = target.dataset.src;
  });
}

#8 Theme Customizer and Theme Switcher seem to conflict

Symptom: You have both a Theme Switcher (light/dark toggle) and a Theme Customizer (color/font picker) on the same page. Clicking one seems to reset the other.

This is by design: Both components share the vanduo-theme-preference localStorage key for the light/dark/system mode. The Theme Customizer also stores independent keys for primary color, neutral palette, radius, and font family.

How they coordinate: Theme Switcher only toggles data-theme. Theme Customizer reads the current theme and applies color/font overrides on top. If you change the theme via Switcher, Customizer re-applies its stored color/font overrides. This is intentional — users expect their brand color to persist even when switching between light and dark.

#9 Scrollspy highlights the wrong section

Symptom: Clicking a sidebar link scrolls to the right place, but the active highlight in the sidebar is on a different item.

Most likely cause: The href on your sidebar links doesn't exactly match the id of the target section.

<!-- Sidebar link -->
<a href="#installation" class="vd-nav-link">Installation</a>

<!-- MUST match exactly -->
<section id="installation">
  <h2>Installation</h2>
</section>

<!-- WRONG: href is "install" but id is "installation" -->
<a href="#install">...</a>  <!-- does NOT match id="installation" -->

Also check: Are multiple sections visible in the viewport at once? Scrollspy uses IntersectionObserver and activates the most visible section. If your sections are very short, increase their min-height or add padding so only one is dominant at a time.

#10 Forms look unstyled

Symptom: Inputs, checkboxes, and select dropdowns look like browser defaults — no Vanduo styling.

Most likely cause: Missing the required classes. Vanduo doesn't style bare <input> elements — you must add .vd-input (or .vd-form-group wrapper) and use the specific markup patterns for checkboxes/radios/switches.

Unstyled

No classes = browser default styling.

Styled

Add .vd-input and wrap in .vd-form-group.

Check the Forms component page for the exact markup patterns for checkboxes (.checkbox wrapper), radios (.radio wrapper), switches (.switch wrapper), and ranges (.range class). Each requires specific child elements that Vanduo's CSS targets.

Debugging Techniques

Inspecting CSS Variables in DevTools

Not sure which custom property is applying to an element? Chrome DevTools can show you every active variable:

  1. Right-click the element → Inspect
  2. Select the element in the Elements tree
  3. Open the Computed tab (next to Styles)
  4. In the filter box, type -- — this shows every CSS custom property currently computed for that element
  5. Look for the variable you overrode (e.g., --color-primary). If it shows the default instead of your value, your CSS isn't loaded or isn't winning the cascade.

Finding the Right Class Name

If you're not sure what class a component needs, there are two quick ways to find out:

DevTools Console

Inspect an existing Vanduo element and read its classes:

document.querySelector('.vd-btn').classList
// DOMTokenList(3) ['vd-btn', 'vd-btn-primary', 'vd-btn-sm']
Grep the Framework

Search the CSS source for patterns:

# Find all button variants
grep -r "\.vd-btn-" css/components/buttons.css

# Find all card modifiers
grep -r "\.vd-card-" css/components/cards.css
